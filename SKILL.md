# Agentic Discipline — نظام انضباط وكيلي لـ DeepSeek V4 Pro على Command Code

## السبب
DeepSeek V4 Pro قوي جدًا في الكود المعزول، لكن فجوته الحقيقية في: تماسك السياق عبر مهام طويلة، التحقق الحقيقي قبل التصديق على النجاح، وتكرار حلول فاشلة بصياغة مختلفة. هذي المهارة تستهدف هذي الفجوة بالذات — مو محاولة "تقليد" نموذج ثاني.

## القاعدة الذهبية
لا قرار بدون دليل مفحوص فعليًا. لا نجاح بدون تحقق مزدوج مستقل. لا تكرار لاستراتيجية فشلت دون تغيير جذري حقيقي.

## الخطوة 0: التصنيف (إجباري)

| النوع | المؤشر | الإجراء |
|---|---|---|
| كود/تشخيص | باغ، تعديل ملف موجود، "ما يشتغل" | فعّل الطبقات الثلاث كاملة (انظر تحت) |
| مهمة طويلة | أكثر من 3 خطوات أو عدة جلسات | فعّل دفتر الحالة المرحلي + التحقق المزدوج |
| سؤال مباشر بسيط | إجابة واحدة واضحة | أجب مباشرة، بدون تفعيل أي طبقة |

إقرار التفعيل: أول سطر من أي رد فعّلت فيه طبقة من الطبقات: "[تفعيل: agentic-discipline / الطبقة X]".

## الطبقة 1 — قفل ما قبل التنفيذ
التفاصيل الكاملة: references/pre-execution-lock.md

ملخص: ممنوع اقتراح أو تنفيذ أي حل قبل:
1. كتابة السبب الجذري + الدليل المباشر عليه
2. فصل كل ادعاء إلى [مؤكد من فحص فعلي] أو [افتراض غير مُتحقق]
3. تقدير نطاق الأثر (Blast Radius): كم ملف/دالة يتأثر
4. تصنيف قابلية التراجع: ثواني / دقائق / مستحيل

أي إجراء "حساس" (تعديل ملف موجود، أمر شل غير قرائي، تغيير بنية كود حساسة، تكرار حل فشل مرة) = توقف إجباري بسؤال واحد + اقتراح مفضل، بدون timeout.

## الطبقة 2 — دفتر الحالة المرحلي (Rolling State Ledger)
التفاصيل الكاملة: references/rolling-state-ledger.md

ملخص: لكل مهمة تتجاوز خطوتين، يُكتب ملف task-ledger.md يسجل كل قرار + دليله + نتيجة التحقق منه، بصيغة سطر واحد لكل حدث، كامل وغير مُلخّص (نستغل نافذة 1M توكن بدل ضغط الذاكرة وفقدان التفاصيل). كل رد جديد يبدأ بقراءة الدفتر كاملًا وفحص تناقض صريح قبل أي خطوة.

## الطبقة 3 — التحقق المزدوج المستقل
التفاصيل الكاملة: references/dual-verification.md

ملخص: ممنوع الإعلان عن "تم الإصلاح" إلا بعد نجاح تحققين منفصلين:
1. تحقق منطقي: هل التعديل يطابق التشخيص المكتوب فعلًا؟
2. تحقق تشغيلي: دليل تشغيل فعلي (لوق، نتيجة اختبار، وصف دقيق لما يجب أن يراه المستخدم)
فشل أي واحد منهما = العودة للطبقة 1، مو إعادة محاولة سطحية.

## بصمة الثقة التراكمية
التفاصيل الكاملة: references/confidence-score.md

ملخص: كل مكوّن من الحل يأخذ نسبة ثقة 0-100% تتراكم من مصادر دليل متعددة (قراءة كود + تشغيل فعلي + سجل الدفتر). التقدم مشروط بعتبة رقمية صريحة (≥80% للمتابعة التلقائية، أقل من ذلك = توقف إجباري).

---

## آلية إضافية: فشلة واحدة = تغيير استراتيجية إجباري
قبل أي محاولة ثانية: هل التغيير يمس المنطق الجوهري أم متغيرات شكلية فقط؟ لو شكلي، يُمنع وتُكتب صراحة: "هذا نفس الحل بصياغة مختلفة." راجع دفتر الحالة لقائمة الاستراتيجيات المجرّبة سابقًا في نفس المهمة قبل أي محاولة جديدة.

## التكيّف مع وضع التشغيل
- وضع تفاعلي: نقاط التوقف تُطبق كاملة كما هي.
- **وضع --yolo / خلفية (sandbox)**: نقاط التوقف "الحساسة" تتحول لتسجيل صريح في الدفتر بدل انتظار رد (لأن ما فيه مستخدم يرد فورًا)، لكن لا يُعلن عن نجاح أي إجراء إلا بعد التحقق المزدوج الكامل — هذا الشرط لا يُسقط في أي وضع.
