# دفتر الحالة المرحلي (Rolling State Ledger)

## الفكرة الجوهرية
بدل تلخيص الذاكرة (اللي يفقد تفاصيل ويسبب context drift)، نستغل نافذة سياق DeepSeek V4 Pro الضخمة (1M توكن) لنحتفظ بسجل **كامل وغير مُلخّص** لكل حدث في المهمة. هذا يحل مشكلة فقدان التماسك من جذورها.

## متى يُنشأ
أي مهمة تتجاوز خطوتين، أو مهمة يُتوقع امتدادها عبر أكثر من رد/جلسة.

## الملف: `task-ledger.md`

### البنية (سطر واحد لكل حدث، صيغة ثابتة)
```
[رقم] | الوقت/الخطوة | نوع الحدث | المحتوى | الدليل | نتيجة التحقق
```

أنواع الأحداث:
- `DIAGNOSIS` — تشخيص سبب جذري
- `DECISION` — قرار تنفيذ + البدائل المرفوضة وسبب رفضها
- `ACTION` — إجراء فعلي تم تنفيذه
- `VERIFY` — نتيجة تحقق (منطقي أو تشغيلي)
- `FAIL` — محاولة فشلت + سبب الفشل بالضبط
- `STRATEGY_USED` — وصف الاستراتيجية المستخدمة (لمقارنة المحاولات القادمة بها)

### مثال
```
001 | خطوة 2 | DIAGNOSIS | getPlot يبحث في item.plot بدل item.info.plot | فحص console.log سطر 42 | [مؤكد]
002 | خطوة 2 | DECISION | تعديل ترتيب البحث في getPlot | البديل المرفوض: إعادة هيكلة كامل item — مرفوض لاتساع النطاق | -
003 | خطوة 3 | ACTION | تعديل دالة getPlot في plot-utils.js | - | -
004 | خطوة 3 | VERIFY | تشغيل الاختبار test_getPlot.js | لوق: 5/5 ناجح | [مؤكد تشغيليًا]
```

## القاعدة الإجبارية لكل رد جديد بنفس المهمة
1. اقرأ الدفتر **كاملًا** من البداية (مو ملخص، مو آخر 3 أسطر فقط)
2. قارن آخر إجراء/قرار مع كامل السجل — هل فيه تناقض؟
3. لو وُجد تناقض: توقف فورًا واذكره صراحة قبل أي خطوة جديدة
4. أضف الحدث الجديد بنفس الصيغة فور حدوثه — لا تأجيل التسجيل لنهاية الرد

## الفرق عن `task-state.md` التقليدي
- لا تلخيص أبدًا — كل حدث يُكتب كاملًا بدليله
- لا "آخر قرار" فقط — السجل الكامل متاح دائمًا للمراجعة
- يُستخدم أيضًا كمصدر لقائمة الاستراتيجيات المجرّبة (راجع `STRATEGY_USED` قبل أي محاولة جديدة بعد فشل)

## بصمة الخطأ الدقيقة (Error Fingerprint)

### الهدف
رفع القوة الفعلية للنموذج (خصوصًا بوضع High) عبر منعه من إعادة اكتشاف نفس الخطأ التقني الدقيق بالتجربة والخطأ — لا تقليل عمق تفكيره، بل بناء كل تفكير جديد فوق معرفة تراكمية حقيقية.

### نوع حدث جديد بالدفتر: `FINGERPRINT`
يُسجَّل فور اكتشاف أي خطأ تقني **دقيق وقابل للتكرار** (bit manipulation، قيمة hex خاطئة، صيغة API محددة، ترتيب معاملات، إلخ) — مو أخطاء عامة، بل التفاصيل الحرفية الدقيقة اللي يسهل الوقوع بها مرة ثانية.

### الصيغة
```
FINGERPRINT | [المجال التقني] | [الخطأ الدقيق بالضبط] | [الإصلاح الصحيح الدقيق]
```

### مثال حقيقي
```
FINGERPRINT | MPEG-TS PTS bit-mask | byte2 استخدام 0xFE بدل 0x7F عند قراءة marker bit → يفسد PTS[21:15] | الصحيح: قناع 0x7F مع >>1، ماركر بت لا يُقرأ كجزء من PTS data
FINGERPRINT | MPEG-TS PTS bit-mask | استخدام 0xF0 بدل 0x20/0x30 لبتات 7-4 يخلي كل البتات 1 | الصحيح: القناع يحدد فقط البتات المطلوب تعيينها، لا يصفّر الباقي بالخطأ
```

### القاعدة الإجبارية
قبل كتابة أي كود يلمس نفس المجال التقني (bit manipulation، بروتوكول محدد، مكتبة معينة)، **افحص كل سطور `FINGERPRINT` بالدفتر أولاً** — حتى لو كانت من محاولة سابقة فشلت بالكامل وتم التراجع عنها بالمشروع. الفشل السابق لا يُمحى من الذاكرة التقنية حتى لو أُلغي من الكود.

## الاسترجاع الفوري بعد الانقطاع (Ledger-First Recovery)

### الهدف
تحويل أي انقطاع (انهيار ذاكرة، إغلاق مفاجئ، بداية جلسة جديدة لمهمة قائمة) من "إعادة بداية من الصفر" إلى "استكمال دقيق من نقطة التوقف" — بنفس عمق التفكير (High)، بدون إعادة استكشاف مكرر يستهلك تكلفة بلا داعي.

### القاعدة الإجبارية (أول إجراء في أي جلسة جديدة لمهمة سبق أن بدأت)
1. **قبل** فتح أي ملف كود أو بدء أي استكشاف جديد: اقرأ `task-ledger.md` كاملًا
2. حدد بدقة: آخر `ACTION` ناجح، آخر `FAIL` وسببه، كل سطور `FINGERPRINT` المسجّلة، وآخر `DECISION` معلّق بدون `VERIFY`
3. استأنف العمل من هذي النقطة مباشرة — لا تُعِد قراءة ملفات سبق توثيق محتواها بالدفتر بما يكفي للمتابعة
4. لو الدفتر يوضح إن مسارًا معينًا فشل مرتين بنفس `FINGERPRINT`: لا تُعِد نفس المسار، انتقل مباشرة للبديل الموثق بـ`next steps` إن وجد، أو اعرض على المستخدم البدائل المتاحة

### الفرق عن سلوك افتراضي بدون هذي القاعدة
بدون الدفتر: انقطاع = بداية من الصفر = إعادة استكشاف كامل بنفس العمق (تكلفة مضاعفة بلا قيمة إضافية).
مع الدفتر: انقطاع = قراءة سريعة لملف واحد = استئناف فوري من نفس نقطة التوقف بكامل القوة، بدون إهدار.
