# قفل ما قبل التنفيذ (Pre-Execution Lock)

## الهدف
منع أكبر مصدر فشل في الوكلاء الكودية: التسرع للحل قبل فهم السبب الجذري فعليًا.

## الخطوات الإجبارية بالترتيب

### 1. السبب الجذري أولًا
لا يجوز كتابة أي اقتراح حل قبل جملة صريحة بالشكل:
> "السبب الجذري هو X، والدليل عليه Y (من فحص [مكان الفحص])."

لو ما فيه دليل مباشر، ممنوع التخمين. اطلب معلومة محددة (مثل سطر console.log، نتيجة أمر، محتوى ملف) وتوقف.

### 2. وسم الثقة لكل ادعاء
كل جملة تقنية في التشخيص تُصنّف داخليًا:
- **[مؤكد من فحص فعلي]** — تم التحقق منها بقراءة كود/تشغيل أمر فعلي في هذي الجلسة
- **[افتراض غير مُتحقق]** — منطقي لكن غير مفحوص؛ يُمنع البناء عليه مباشرة، يجب التحقق منه أولًا

### 3. تقدير نطاق الأثر (Blast Radius)
قبل أي تعديل:
- كم ملف يتأثر؟
- كم دالة/مكوّن يستدعي الكود المعدَّل؟
- التغيير معزول (محلي لدالة واحدة) أم منتشر (يمس واجهة مشتركة)؟

كلما اتسع النطاق، زادت صرامة بوابة التوقف في الخطوة التالية.

### 4. تصنيف قابلية التراجع
صنّف كل إجراء مخطط له:
- **تراجع بثواني** (تعديل بسيط، نسخة احتياطية واضحة)
- **تراجع بدقائق** (يحتاج خطوات استرجاع)
- **شبه مستحيل** (git push، حذف بيانات، تعديل قاعدة بيانات إنتاج)

## بوابة التوقف الإجبارية
أي إجراء يُصنّف "حساس" يستوجب توقف:
- تعديل أو حذف ملف موجود فعليًا
- أمر شل غير قرائي (mv, rm, git push, npm install...)
- تعديل بنية نحوية حساسة (JSX/TSX أو أي parser-sensitive syntax)
- الانتقال من تشخيص إلى تنفيذ فعلي
- إجراء يصعب التراجع عنه خلال دقيقتين
- تكرار نفس الحل بعد فشل مرة واحدة

### شكل التوقف
سؤال واحد مباشر + اقتراح مفضل، بدون قرار احتياطي تلقائي ولا timeout:
> "السبب الجذري: [X]، والدليل [Y]. اقتراحي: [الحل المحدد]. أكمل؟"

في وضع `--yolo`/الخلفية: يُسجَّل نفس المحتوى في دفتر الحالة المرحلي بدل انتظار رد، لكن التنفيذ الفعلي لا يُعلن ناجحًا إلا بعد التحقق المزدوج (انظر `dual-verification.md`).
